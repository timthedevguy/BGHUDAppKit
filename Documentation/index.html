Introduction

The missing OS X Leopard/Snow Leopard HUD controls.

BGHUDAppKit is a small framework I designed to fill the gap left by Apple when they released the nifty HUD window but forgot to release their controls that match it. There are few really good frameworks out there that will provide you with the HUD style interface. This is where the big change in BGHUDAppKit becomes clear. I use absolutely NO resources or images to give these controls the HUD appearance. BGHUDAppKit uses 100% native Cocoa drawing functions like NSColor, NSGradient and NSBezierPath. Because of this the framework is much smaller, provides a cleaner interface and is already Resolution Independence ready. And of course it is free and open-source.

Project Install

1
Right-Click on your Frameworks group in XCode.

2
Select Add -> Existing Frameworks...

3
Navigate to your BGHUDAppKit build/Release directory and select BGHUDAppKit.framework (Yes, it's a folder)

4
Right-Click on your projects Target (This is generally a target with the same name as your project) located in your Targets group in XCode

5
Select Add -> New Build Phase -> New Copy Files Build Phase

6
Double-Click the new item named Copy Files under your projects target grouping. (You may need to expand your projects target to see this)

7
Select Frameworks from the Destination drop down, close the window when your done.

8
Drag BGHUDAppKit.framework from Frameworks group in XCode to the Copy Files build phase under Targets group in XCode.

9
Build your project.

You should now be able to inspect your products .APP bundle and see a new folder named Frameworks that contains BGHUDAppKit.

Usage

Usage is now as simple as opening your MainMenu.nib and dragging a control from the BGHUDAppKit palette inside of Interface Builder.

BGHUDAppKit controls will look exactly like they do at runtime while you are designing your interface in IB. The only exception to this is when custom themes are in use. Those custom themes will not show up in design time since the BGThemeManager has no idea they exist. See more on Themeing below.

Themeing

BGHUDAppKit is more powerful than it first appears to be. I have designed the classes with the mentality of reinvention. Why should everyone have to reinvent the NSScroller if they want to apply iTunes style colors to it? You shouldn’t have to. I’ve done the hard work and research for you. My purpose in life as a developer is to make life easier, programmers are lazy :) Every control in BGHUDAppKit uses a theme, you can have one theme for every control or you could have 10 different themes for 10 different controls.

1
Creating a theme does take a little bit of work. If you look inside the Headers directory of BGHUDAppKit.framework you will see a file named BGTheme.h. This is the master theme file. Inside of it you will see every theme method available and a short description of each explaining where it is used at. Just create a subclass of BGTheme and start overriding whatever methods you need to change.

2
BGHUDAppKit uses a singleton BGThemeManager object. This makes it so that there is only ever one instance of the theme allocated, thus saving memory. In this example my subclassed theme is myTheme.m. This code is in my main AppController object.

-(void)awakeFromNib {
  [[BGThemeManager keyedManager] setTheme: 
    [[[myTheme alloc] init] autorelease] 
                                   forKey: @"myTheme"];
}

What’s happening? We are telling BGThemeManager keyedManager object that we want to add a new theme instance to its memory with a key of myTheme. You can add as many as you like. Two default themes are provided for you. The one you’ll see when you Drag’n’Drop from the IB palette is gradientTheme, this is also the theme you will always see when you simulate the interface in IB. This is because your custom theme object doesn’t exist yet. The second default theme is flatTheme, this theme is just a slight dark gradient, giving the controls a less candy look.

3
Making the controls use this theme is even easier. In IB a new inspector panel will present itself when you are in the proper focus for that control. Some controls, like buttons, have this panel in the Cell focus. Click on a button once, and then again and it will get a halo around it. You will see a new section called Theme with a text box labeled Theme Key. Just type your theme key in there and when you build the app that control (and only that control) will use your custom theme.

Shrinking

BGHUDAppKit is appox 1.0 Mb in size. This is quite large for a framework this simple. The extra disk space is taken up by the BGHUDAppKitPlugin which is the IB Plugin. This really doesnâ€™t need to be in your distributed app, the framework will function normally without it. Do NOT delete BGHUDAppKitPlugin from your source BGHUDAppKit.framework.

The reason the framework is built this way is because IB will automatically look in a linked framework for a plugin. This seemed, to me at least, the most logical and easiest way to distribute the plugin to you the developer.

A way to shrink your final APP size is really simple. Create a new Run Script build phase and copy the following line to it.

rm -fR "${CONFIGURATION_BUILD_DIR}/${WRAPPER_NAME}
/Contents/Frameworks/BGHUDAppKit.framework/Versions/
Current/Resources"
Note: The above line is wrapped to fit in this window, please ensure it's a single line when you paste it

The next time you build your project XCode will look in your final .APP folder and remove the unneeded Resources directory that BGHUDAppKitPlugin resides in. This shrinks BGHUDAppKit.framework down to approx 400k. Quite a big difference.

Getting Help

You may email me on GitHub for assistance. So far the majority of users have found no problems getting started. Please report any bugs you find on the GitHub issue tracker page.

Known Issues

BGHUDAppKit is currently incompatable with XCode 4. It will compile and run, but some controls are drawing badly. Use it at your own risk. I will be upgrading the code as I go.

Radio button matrices are flawed due to the odd way that BGHUDAppKit has to determine if the underlaying NSButtonCell is a Radio or Check. Pre App Store this worked fine, but with the removal of private variable usage it will remain broken until a better solution can be found.
